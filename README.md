# yeeap-charity-donation · 公益捐赠 Skill

> 当用户表达「我想捐助」「慈善捐助」「公益捐赠」「我要捐款」「献爱心」等意向时，引导用户选择公益项目，通过 **yeeap 钱包**完成扣款，同步捐款记录到公益平台，并生成**捐赠证书**展示给用户。

业务侧 Skill，基于 yeeap 钱包支付能力构建。支付环节由官方 **`yeeap-wallet`** Skill 完成，本 Skill 只负责「下单 + 履约（同步捐款 / 取证）」。

---

## 一、端到端流程

```
用户表达捐助意向
        │
        ▼
┌─────────────────────────────────────────────────────────┐
│ 阶段一  浏览下单                                          │
│   projects.py    → 查询公益项目列表（后端固定条件）        │
│   create_order.py → 创建捐赠订单，写入本地订单文件          │
└─────────────────────────────────────────────────────────┘
        │  order_no + app_id
        ▼
┌─────────────────────────────────────────────────────────┐
│ 阶段二  支付（官方 yeeap-wallet Skill）                  │
│   传入 order_no + app_id，由 yeeap-wallet 扣钱包余额      │
│   支付成功后把支付凭证写回订单文件                        │
└─────────────────────────────────────────────────────────┘
        │  订单文件含支付凭证
        ▼
┌─────────────────────────────────────────────────────────┐
│ 阶段三  同步取证（履约）                                  │
│   service.py → 后端解密凭证 → 同步捐款记录 → 拉取捐赠证书  │
└─────────────────────────────────────────────────────────┘
        │
        ▼
   展示捐赠证书链接给用户
```

三阶段强依赖，不可跳过。详见 [`SKILL.md`](./SKILL.md)（Agent 执行规范）。

## 四、履约幂等与失败重试（重要）

支付成功后进入履约（阶段三），若同步或取证失败，**支持重试且不会重复扣款**：

- **不会重复支付**：`service.py` 仅读取订单文件里的支付凭证完成履约，后端全程不会再调用支付、不碰钱包。
- **不会重复同步捐款**：后端记录该订单是否已同步捐款，首次同步成功后标记；重试时跳过同步、**只重新拉取证书**，避免产生重复捐款记录。
- **证书可反复拉取**：取证接口幂等，`CERT_STATUS=GENERATING` 可稍后重试。
- **重试方式**：用**同一个 `orderNo`** 重跑 `service.py`，间隔 3~5 秒、最多 3 次。
- **严禁**因履约失败而重新创建订单或重调 `yeeap-wallet`（会重复扣款）。

> 履约重试需在支付后一定时间内完成（订单上下文有效期 1 天）。

## 五、安全约束

- 支付凭证任何形式**不得回显给用户、不得写入业务日志**。
- `skill_id` 必须由 YEEAP 平台登记后下发，不得本地伪造。
